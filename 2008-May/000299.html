> 
> I think that the software author should be responsible for i18n. He/she should 
> provide framework for managing translations. We all know that this is not how it 
> works in real world. In cases when l10n is not a part of the program build 
> process all we can do is to provide a patch with translated .desktop file.
> For such project I would recommend just to add their .desktop file to our tree 
> and run make_intltool_update script. If we need the true blue .desktop file with 
> translations for such project (we need to prepare patch for SRPM) - just launch 
> script build_desktop_files.
> If, on the other side, we can change the software build process to some extent, 
> the best way is to have .desktop.in file as a source and to build a .desktop 
> file from it and from the .po files during the build process. All you need for 
> this is the intltool-merge utility (see VIT SRPM for example).
> Mature projects usually already have such .desktop file compilation in their 
> build process.
> The important thing is that .desktop.in file is the real source file - .desktop 
> file can be "cooked" any time from it if you have translations.
> 
> This is rather complex issue and to find a complete solution I would need to 
> examine the problem a little more and to answer few questions:
> 
> How do the translated .desktop files get into official packages now ?
> Isn't it too labour-intensive ?
> Should we provide some patch-making infrastructure ?
> Or should we just make one-for-all package from our .desktop files rather than
